Stochastic Track Port — Project Retrospective

May 17–24, 2026 · 95 commits across 3 branches · STM32 Eurorack firmware

1. Project Overview

Objective: Port the Vinx fork's Stochastic track type to the XFORMER performer module, then extend it into a full generative sequencer with ticket-based probability shaping, Marbles-style distribution control, burst/ratchet support, loop/live source modes, and a deterministic lock/freeze mechanism — all within an STM32F405RGT (128 KB SRAM) budget.

Scope: A mono generative sequencer track type living alongside existing Note, Curve, Indexed, and Tuesday tracks. Generates CV/gate output from probability-weighted degree and duration tickets, with looper controls (window, mask, tilt, patience, mutate) and per-pattern 16-slot storage. UI surface spans four hero pages (Core/Marbles/Direct/Loop) plus ticket editor pages, all on a 128×64 monochrome OLED.

Timeline: ~7 days from first commit to current WIP branch (feat/stochastic-seed-log). Implementation is ongoing — the storage architecture redesign (Phase 14B seed+log, Phase 16 flat-cell, Phase 17 sequence reduction) is in progress.

2. Key Mistakes & Challenges

2.1 — Heap Allocation Crash on STM32 technical

Each StochasticTrackEngine constructor called new LockedParentEvent[64], allocating 5,888 B per track. The STM32F405 has ~12 KB free between end-of-.bss and top-of-stack, shared by heap and stack. Two stochastic tracks consumed 11,776 B — leaving 52 B for the call stack. The ARM _sbrk implementation does not check against the stack pointer, so new returned memory that silently overlapped the live stack. The firmware hardfaulted on reboot with no warning. The fix deleted the heap allocation outright and deferred the Lock feature redesign.

Root cause: Assuming host-environment heap semantics apply on a bare-metal MCU. The allocation did not overflow .bss or trigger a linker error — it consumed the dynamic headroom that the stack also needed.

2.2 — Host vs Target sizeof Mismatch technical

Multiple features were RAM-costed using host-compiled sizeof estimates rather than ARM sizeof probes from the STM32 release build. Container alignment, Routable<> storage overhead, and compiler padding differ between x86 and ARM Cortex-M. Several design decisions (e.g., "we have room for this field") were made against incorrect numbers. PROJECT.md now mandates STM32 sizeof probes as the source of truth.

2.3 — Accent / Patience Storage Alias technical

Two independent engine semantics — accent probability and patience-melody — shared one byte (_accentProb). Setting patience M to 100 (the "off" sentinel) silently caused every generated event to carry accent=true. The accent flag was never consumed by the audio path, so the bug was invisible at the output but corrupted downstream logic that depended on the byte's value. Fixed by killing the accent semantic entirely and renaming the field to _patienceMelody.

Root cause: Field repurposing without grep-auditing every read site. The rename from _accentProb to _patienceMelody was only cosmetic — the real fix required understanding who reads the byte and for what purpose.

2.4 — Track vs Sequence Ownership Confusion technical

Phase 9 (V4 ownership correction) discovered that pitch tickets, rhythm/melody source modes, generator controls (complexity, rate, variation, rest, burst), evolution knobs (sleep, patience, mutate, jump), and even the loop-window rotate were all Track-owned. This meant switching patterns 1–16 did not recall different harmonic masks, rhythm styles, or evolution rules. A pattern was just an arbitrary 64-step loop window over global track settings — not a self-contained musical snapshot. The fix moved ~29 fields from Track to Sequence (16× multiplier on RAM), requiring a 48-bit event packing pass to stay under the NoteTrack container gate.

Root cause: The original Vinx port inherited a Track-owns-everything topology. No one asked "where does state live?" before building the engine. AGENTS.md lists this as the first invariant but it was skipped during the port.

2.5 — Serialization Symmetry Gaps technical

StochasticSequence::clear() and sanitizeAfterRead() knew about _density, _durationTickets[], and _level — but write() and read() didn't. PHASE9-V4 explicitly warned that read/write order must remain symmetric. The gap was found during a systematic debugging pass (Phase 10.6 bug catalog), not during implementation. Also: StochasticTrack::_lock was serialized despite being declared engine-only (runtime transient) in the ownership table.

Root cause: Fields were added to clear/sanitize without updating write/read. No serialization round-trip test existed to catch the asymmetry.

2.6 — Burst Gate Scheduling: Parent OFF Masks Children technical

The parent gate-OFF fired at tick+10 (50% of a 1/16 parent), cutting any child whose ON landed before that offset. With 4 children at burstRate=50 in a 48-tick parent, child offsets were approximately 4, 9, 14, 19. Child 2 (offset 9) fired ON at tick+9, then the parent OFF at tick+10 forced it low — a 1-tick gate effectively lost. Two of four children were silently truncated. The fix skipped the parent OFF push entirely when children exist, letting the last child's OFF be the natural gate-low.

Root cause: Gate scheduling with nested events requires ordering analysis. The simple formula "parent gate = min(50% of duration, 10 ticks)" didn't account for children whose ON falls inside that window.

2.7 — Integer-Division Cliffs in Continuous Knobs technical

Five places used KNOB / N integer division for 0..100 UI knobs mapped to small ranges. Knob values 1–2 silently produced zero effect (e.g., sleep knob at 2 produced 0 sleeps). Found in Variation, Complexity, Marbles Spread, Sleep, and Contour drift. Fixed uniformly by multiplying the triangle term by 10 before division. Also: (sleep × 4 + 5) / 10 (round, was floor) so knob=2 → 1 sleep instead of 0.

Root cause: Using integer math for continuous knob ranges without checking the quantization floor. The pattern was copy-pasted across multiple pickers.

2.8 — Feature Scope Drift & Architectural Churn communication

The project went through V2 → V3 → V4 → V5 control granularity models, then Phase 11 simplification (collapsing 3 levels into flat list), Phase 12 follow-ups, Phase 13 feature planning, Phase 14 trunk vs seed+log, Phase 14B seed+log, Phase 15 pitch math review, Phase 16 flat-cell model, and Phase 17 sequence reduction. Each phase reworked the same surface area. 17 task-wiki documents span hundreds of lines, with cross-references marking older docs as "superseded." Commit messages reference phase numbers (P9, P11, Phase 16 P7) that are meaningless without the task wiki.

Root cause: Storage architecture (what gets stored vs what re-rolls, identity-as-output vs identity-as-recipe) was not decided before building the UI and engine layers. The design kept oscillating between "baked event tape" and "seed + knobs" models. No one asked "what is the identity of a pattern?" before writing code.

2.9 — Adversarial Review Ran Too Late communication

Codex adversarial reviews flagged real bugs in the mutation pipeline, Mask/Tilt window scoping, and musicality cliffs AFTER Phase 12 landed. The fixes required reworking mutation targets, mask rank generation, burst divisor math, and continuous-knob math — all post-merge. Had the adversarial pass run continuously during implementation, several of the integer-division cliffs and window-scoping bugs would have been caught before they shipped.

2.10 — Reference Vocabulary Leak communication

Terms from Proteus ("density", "complexity"), Marbles ("spread", "bias", "steps", "deja-vu"), Melodicer ("rate", "variation", "dice/realtime"), Tuesday ("burst"), and SIG ("tickets") leaked into code and documentation. The Phase 7 dictionary was created mid-project as a corrective — translating external vocabulary to Performer-native concepts. By that point, code already used the borrowed terms. The dictionary itself accumulated amendations as design decisions shifted (e.g., "divisor-relative" vs "absolute tick" duration tickets).

3. Lessons Learned

Technical

#LessonTrigger
1Never heap-allocate on a bare-metal MCU without proving the stack margin. Pre-allocate via file-scope BSS arrays or linker-placed sections. If you must heap-allocate, verify _sbrk checks against the stack pointer — many embedded libc implementations don't.Heap crash (2.1)
2Run STM32 release sizeof probes as the source of truth for RAM costing. Host sizeof differs in alignment, padding, and template instantiation. Every feature-gate decision must be made against ARM numbers.Host vs target sizeof (2.2)
3Never alias two independent semantics onto one storage byte. If you repurpose a field, grep every read site and verify none of them depend on the old meaning. Renaming the C++ identifier is cosmetic — the bytes are positional in serialization.Accent/patience alias (2.3)
4Establish ownership contracts before implementation. Ask "where does state live?" for every field before writing the first line of engine code. Track-owns-everything is the wrong default for pattern-based sequencers.Track vs sequence ownership (2.4)
5Write serialization round-trip tests before implementing serialization. A test that writes, clears, reads back, and asserts equality catches every field-mismatch bug immediately. Without it, the gap between clear/sanitize and write/read grows silently.Serialization symmetry (2.5)
6Gate scheduling with nested events requires ordering analysis. Draw the timeline of parent gate-ON, parent gate-OFF, and each child's ON/OFF before implementing. A simple "parent OFF at 50% duration" rule breaks when children fire inside that window.Burst gate masking (2.6)
7For 0..100 UI knobs mapped to small ranges, multiply before dividing. KNOB / N integer division creates dead zones at low knob values. Scale the numerator by 10 (or use floating point) so the quantization floor doesn't swallow the first few knob positions.Integer-division cliffs (2.7)
8Design the storage model and identity contract before building UI and engine layers. Oscillating between "baked event tape" and "seed + knobs" rewrites the entire system each time. The identity question ("what makes a pattern a pattern?") must be answered first.Scope drift (2.8)

Communication

#LessonTrigger
9Use task-oriented naming, not numbered milestone tracking for rapid iteration. Phase numbers (P9, P11, Phase 16) leak into commits and become cognitive overhead. They imply a linear progression that doesn't match the actual design oscillation. Name documents by what they decide, not where they fall in a sequence.Scope drift (2.8)
10Aggressively consolidate documentation. Every doc that says "See also X, which supersedes Y" is a navigation tax. After a design shift, merge the surviving decisions into one current document and mark the old ones as historical appendices — don't leave a chain of partially-outdated references.Documentation debt (2.8, 2.10)
11Run adversarial review continuously during implementation, not as a post-landing gate. Codex reviews caught real bugs (mutation pipeline, window scoping, musicality cliffs) after Phase 12 landed. Integrating review into the development loop would have caught several of these before merge.Late adversarial review (2.9)
12Translate external reference vocabulary to project-native terms before coding begins. Borrowing Proteus/Marbles/Melodicer names for knobs and code identifiers creates a layer of indirection. The names carry assumptions from their original systems. Define a project-native vocabulary first; reference external modules in design notes, not in user-facing labels or code.Vocabulary leak (2.10)

4. Recommended Actions

Technical — to prevent recurrence in future projects

R#ActionPrevents
R1RAM gate checklist for every feature branch. Before merging: build STM32 release, run arm-none-eabi-size, check .data + .bss against the hard-warning zone (120 KB), check .ccmram_bss, and measure sizeof for the changed track/engine types. Record the numbers in the commit message or a status file.Heap crash, sizeof mismatch (2.1, 2.2)
R2No heap allocation in engine constructors. Add a lint rule or review gate that flags new / malloc in any file under src/apps/sequencer/engine/. Pre-allocated file-scope arrays (like gDirectHistory) are the established pattern.Heap crash (2.1)
R3Serialization test as part of model implementation. Any write()/read() pair must have a corresponding test that writes known values, clears, reads back, and asserts equality. Run this test on every commit that adds or reorders a serialized field.Serialization symmetry (2.5)
R4Ownership audit as a design gate before engine code. Before writing triggerStep or generateRhythm, produce a one-page table answering "sequence-owned or track-owned or engine-only?" for every field. Review it against the pattern-switch use case. The table from PHASE9-V4 is the template.Track vs sequence confusion (2.4), field aliasing (2.3)
R5Gate scheduling test with multiple burst children. Add a test that generates a parent with 4 children at short duration (e.g., 48 ticks) and asserts every child's gate-ON precedes its gate-OFF and no child's gate is truncated by the parent's gate-OFF. Run on every commit that touches burst or gate logic.Burst gate masking (2.6)
R6Continuous knob math review. For any 0..100 knob that drives a range smaller than 10, verify with a table: knob=0 → output, knob=1 → output, knob=2 → output, knob=50 → output, knob=100 → output. Flag any dead zone where the first N knob positions produce identical output. The Phase 12 musicality fix (×10 scaling) is the pattern to follow.Integer-division cliffs (2.7)

Communication — to prevent recurrence in future projects

R#ActionPrevents
