Stochastic Track - Lessons Learned

Generalizable insights for future firmware projects

Project Overview

Objective: Port Vinx fork Stochastic track to XFORMER, extend into Performer-native probability-driven generative sequencer.

Timeline: Phases 1-16 (May 2026) - V3, V4 ownership, V5 dictionary, Phase 14B seed+log, Phase 16 flat cell.

Constraints: STM32F4 with 128KB SRAM, 64KB CCMRAM. Track gate: NoteTrack=9544B. Engine gate: 912B.

Lessons Learned - General for Future Projects

CategoryLessonConcrete Action
Technical Ownership mapping is DAY 1 for new track types Before writing code, create a 2-column table: Track-owned (global performance state: octave, transpose) vs Sequence-owned (pattern snapshot: anything that changes when switching patterns 1-16). Reference existing track types (Note, Curve, Tuesday) as authority. Deviations must be explicit decisions, not accidents.
Technical RAM gates are architectural constraints, not optimizations At project start, record current container sizes: Track container = max(sizeof(NoteTrack), sizeof(CurveTrack), ...); Engine container = max(sizeof(NoteTrackEngine), ...). New track types must fit under these gates. If they do not, stop and propose architecture changes before implementing features. Heap and indirection are valid answers.
Technical Serialization round-trip test is MANDATORY Write this test pattern for every serializable struct: (1) Set EVERY field to distinct non-default value, (2) Write to buffer, (3) clear() or default-construct fresh instance, (4) Read from buffer, (5) Assert equality for every field. This catches missing fields and read/write order divergence BEFORE hardware sees it.
Technical Integer division creates cliffs - always pre-scale The pattern "knob_value / divisor" creates silent dead zones. Fix: multiply by a factor before division. Audit every "/" operator in parameter mapping code. Prefer floating point for continuous parameters, or explicit x10 scaling for integer kernels.
Communication When renaming a concept, EXPLICITLY supersede the old name Put a clear notice at the TOP of documents that get renamed or superseded: "SUPERSEDED NOTICE: This document uses Density to mean deterministic loop thinning. Current model uses: Mask = deterministic thinning, Density = generator sound/rest amount. See CURRENT-DOC.md for authoritative definitions." Do NOT leave multiple definitions of the same term alive without disambiguation.
Communication Rejected designs need "Why rejected" at the TOP When you decide NOT to take a design path, add a 3-sentence section at the very top: STATUS: Rejected; Considered: Trunk model with baked CV+gate cells; Rejected because: Would require full engine rewrite breaking compatibility; Use instead: Phase 14B Seed+Log model. Readers should not have to consume 257 lines to learn "we are not doing this."
Technical Never import STL usage from reference code Audit all incoming reference code for: std::vector (replace with fixed arrays or explicit heap), std::sort (verify actually needed; often direct weighted sampling is clearer), std::mt19937/normal_distribution (use projects native Random class), std::rand()/time(NULL) (never use in engine code). XFORMER runs on bare metal with FreeRTOS. Standard library is incomplete in nano newlib.
Technical Cache invalidation needs a single owner and audit trail Pattern: Centralize cache invalidation so it is not distributed across every setter. Good: setParameter() calls a single invalidateCache() that the engine observes. Bad: Each setter has its own logic about whether to invalidate. Some forget. This is a bug magnet. Audit pattern: For every knob affecting generation, ask: "If I turn this while locked, should it be ignored? If not, does the edit invalidate the cache?"
Communication One authoritative dictionary - not multiple phase docs Maintain exactly ONE document that defines the vocabulary: every user-facing knob name, every model field name, what is Track-owned vs Sequence-owned, what is serialized vs transient. All other documents REFERENCE this dictionary. Do NOT let V3, V4, V5, P11, P14B, P16 each define the same terms differently.
Technical Architecture iteration has a cost - minimize rewrites This project went through ~6 major engine architectures. Each rewrite invalidates documentation, breaks tests, creates migration edge cases, leaves dead code paths that look like they might matter. Better: Spend more time in design up front mapping to existing architectural patterns. Incremental evolution beats full rewrites when possible.

Recommended Actions - Prevent/Mitigate in Future Projects

CategoryActionWhen to Apply
Technical Create an ownership checklist template Day 1 of any new track type or model feature. Ask: Track-owned or Sequence-owned? Serialized or transient? Engine-owned or model-owned? Answer these before writing code.
Technical Add serialization round-trip test to CI/test suite Automate for every serializable struct. Make it a test pattern that is easy to copy for new types. This should be as automatic as a compile failure.
Technical Create a "division cliff" linter or audit checklist During code review, flag any integer division where numerator is a user parameter (0-100 knob) and denominator is a constant. These almost always need pre-multiplication by 10 or 100.
Communication Add a STATUS header template to all design docs Every design doc must have at the very top: Status (Draft/Active/Superseded/Rejected), Superseded-by (doc reference), Last-updated date. No exceptions.
Communication Maintain a single VOCABULARY.md file One source of truth for all terms. All phase docs and task docs REFERENCE this file, never redefine terms. When terms change, update this one file and mark old references superseded.
Technical Add a "reference code audit" step to porting plans Before porting code from another fork, systematically audit for: STL usage (vector, sort, mt19937, normal_distribution), heap allocation patterns, timer/seed usage (time(NULL), rand()). Identify incompatibilities before writing a single line of ported code.
Technical Design cache invalidation pattern before adding cache Any time you add a cache between model and engine, first design: (1) What events invalidate it? (2) Who triggers the invalidation? (3) What is the single point of entry? Document this in the code comments. Never let invalidation be distributed across 17 different setters.
Communication Archive and clearly mark obsolete/dead-end designs When a design path is rejected, rename the file with a prefix like "DEADEND-" or "REJECTED-". Move to an archive subdirectory if appropriate. Add a clear 1-paragraph explanation at the top of WHY it was rejected and WHAT to use instead.
Technical Measure container sizes BEFORE implementing new track types At the very beginning, write a sizeof probe and measure: sizeof(Track), sizeof(Engine::TrackEngineContainer), sizeof(NoteTrack), sizeof(TeletypeTrackEngine). These are the gates. New types must fit under the current max or you are proposing an architecture change - which requires explicit approval and analysis.
Communication Explicit decision record at major phase boundaries When moving from one major phase to another (V3->V4, V4->V5, V5->Phase11, Phase14->Phase16), write a 1-page decision record: What changed? Why? What is now deprecated? What is the new authoritative source? This prevents "which document is current?" confusion later.
