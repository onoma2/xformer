Stochastic Track Port — Lessons for Future Projects

Source: 6-day implementation, ~120 commits across 17 phases on feat/stochastic / feat/stochastic-seed-log. This document extracts generalizable lessons from that project for application to all future firmware feature work.

1. Project Context

Port a generative melodic sequencer track type from a fork into a RAM-constrained STM32F4 Eurorack sequencer (128 KB SRAM, 64 KB CCMRAM). The project combined step-probability sequencing with global stochastic constraints (Pitch Weight Tables, Marbles-style distributions, Proteus-style patience/mutation, burst scheduling) and a deterministic keyed-RNG engine with captured-event lock replay.

The project went through 17 phases with multiple architecture rewrites driven by discovered constraints. Key outcomes: the feature is hardware-stable, passing 6 test suites, and is the most complex track type in the firmware. The journey surfaced structural process gaps that apply to any embedded feature project of similar scope.

2. Lessons Learned — General Principles

Technical L1. Define state ownership before writing any code. Every data field must have exactly one owner (persistent model, runtime engine, or UI). The owner defines who writes, who reads, and when. If the engine needs to write persistent state, that's a design smell — it should write to its own runtime state instead.

29 pattern-defining fields were initially placed on Track (global) instead of Sequence (per-pattern). Engine wrote generated events back into persistent model via const_cast, creating task-race conditions. Lock replay buffer was not scoped to patterns, replaying stale cache after pattern changes. All found after implementation, all required architectural rework.

Action: Before Phase 1 of any feature, produce an ownership table: every field, its owner, serialization status, and lifecycle. No code starts without this table committed to the task wiki.
Technical L2. Embedded constraints must gate the design from day one, not be discovered incrementally. STL containers, dynamic allocation in tick paths, floating-point structs in CCMRAM, unpacked buffer layouts — these are not implementation details to fix later. They are design constraints that shape the architecture. Reference code that violates them is reference material, not a starting point.

std::vector, std::sort, std::mt19937 were ported from Vinx and had to be fully replaced. An inline lock buffer (64 × 28 B) would have overflowed CCMRAM by 11 KB across 8 tracks — the entire engine approach had to change to heap allocation. Both were documented as risks in the plan but not gated before implementation began.

Action: Run an embedded-compatibility audit on all reference code before porting. Document banned patterns and mandatory replacements. The audit output is a prerequisite for Phase 1 approval.
Technical L3. The first complex user of a platform subsystem will surface latent bugs. Container destructor semantics, engine tick ordering, heap allocation lifecycle — these were stable until the first track type with non-trivial state exercised them. Expect platform bugs when your feature differs structurally from existing features.

Container placement-new without destructor call → heap leak → hard fault after 2–3 track-mode switches. Engine processed clock events before rebuilding containers → hard fault on project clear. Both were generic platform bugs exposed only by the stochastic track's heap-allocated engine state.

Action: When adding the first feature that exercises a subsystem in a new way (heap allocation, non-trivial destructors, tick-path timing), audit that subsystem's code before building on it. Fix latent platform bugs before feature work.
Technical L4. Serialization round-trip tests must be the first test written. Write → clear → read → assert equality across every field. This catches missing fields, wrong order, wrong defaults, and enum clamp issues immediately. Serialization bugs compound silently if not caught early — they only surface when a user saves and reloads.

Three fields were not serialized; one transient field was. Bug catalog documented them, but the fix came 2+ phases later. During the window, all development was built on a broken save/load foundation.

Action: For any new model type, the first test file is a round-trip test. CI gates on it. No feature work proceeds without a passing round-trip.
Technical L5. Validate the core identity model before building anything on top of it. "Is this feature's identity stored output or recipe replay?" is an architecture question, not an implementation detail. Answering it wrong means rebuilding the engine from scratch — multiple times. A short architecture spike comparing candidates saves days of rework.

The project went through stored-tape (Vinx) → parent/child model (Phase 7b) → seed+log model (Phase 14B) → flat-cell model (Phase 16). Each rewrite invalidated previous implementation. A 2-hour spike comparing these three models before any engine code would have saved significant rework.

Action: For any feature with a novel data/identity model, produce a 1-page comparison of at least 2 candidate architectures with RAM estimates, lifecycle semantics, and migration cost. Lock the choice with user sign-off before Phase 1.
Technical L6. Test control-response at low values, not just extremes. Integer truncation (knob / N) produces silent cliffs at the bottom of ranges. These don't show up in "does knob=0 vs knob=100 produce different output" tests. They show up when a user carefully dials in knob=10 and gets the same output as knob=0.

Variation kernel at knob=10 gave 22% center probability instead of expected ~60%. Complexity kernel leak (max 10 vs triangle max 130) was effectively dead. Sleep knob values 1–2 produced zero sleeps. Found by dedicated adversarial review, not by phase testing.

Action: For every continuous knob, include a sweep test at values 0, 1, 2, 5, 10, 50, 90, 95, 99, 100. Assert output changes meaningfully between adjacent test points.
Technical L7. Integration-level concerns (output pipeline, timing, queues) must be tested early on hardware. Unit tests on the generator don't cover gate/CV scheduling, burst timing overlap, queue overflow, or parent-child gate collision. These are system-level bugs that only surface when the full tick-to-output path runs on real hardware at real clock rates.

Parent gate-OFF silently truncated early burst children. Queue overflow had no defined behavior. Burst at short parent durations produced sub-audible mush. All found during hardware verification, all required engine-level rework, none catchable by generator unit tests.

Action: After engine foundations land, schedule a hardware output-verification pass covering gate timing, burst scheduling, queue overflow, and timing edge cases before building UI or advanced features.
Communication L8. Define MVP scope explicitly, freeze it, and enforce the freeze. A "post-MVP gates" section in the task plan is not a scope freeze — it's a wish list. Scope creep happens one reference module at a time. "MeloDICER has X, SIG has Y, Proteus has Z" turns into "we need X+Y+Z" without anyone making a deliberate decision to include all of them.

Initial plan was a Vinx port. Expanded to include Marbles, Proteus patience/mutation, Tuesday burst, SIG probability weighting, MeloDICER direct controls, and a novel keyed-RNG engine. Each was a reasonable addition individually; collectively they produced 17 phases and multiple architecture rewrites.

Action: Write a one-page scope document with explicit "in" and "deferred" lists. User signs off. Any addition after freeze requires a scope-change entry with impact assessment (RAM, time, architecture risk).
Communication L9. Freeze the vocabulary after the first draft. Control names and field semantics are a specification, not documentation that can be updated at any time. Renaming a control means renaming it everywhere: code, dictionary, UI labels, routing targets, tests, and commit messages. Partial renames create a confusing mess where different documents use different names for the same thing.

Density → Mask → Density (repurposed). Character → Complexity. Rate → NoteDuration. Linearity → Repeat. AccentProb → Patience M. MarblesSteps → Steps sieve. Three dictionary rewrites. Agents reading code couldn't determine field liveness without cross-referencing 5+ phase documents.

Action: After the architecture spike, commit a frozen dictionary. Vocabulary changes require a rename-audit commit that updates all references atomically.
Communication L10. Schedule adversarial reviews early — after foundations, not after features. An adversarial review after engine foundations catches state-ownership races, lifecycle bugs, and serialization gaps when they're cheap to fix. A review after 14 phases catches the same issues plus UI type-guard gaps and lock-identity bugs — now requiring architectural changes instead of design adjustments.

The most valuable document produced was the adversarial review (stoch-review.md), but it arrived after Phase 14. The top finding — engine mutates project model state from the engine task — would have been a Phase 2 fix if the review had been scheduled after engine foundations.

Action: Schedule adversarial reviews at two points: (1) after engine/model foundations land, focusing on state ownership, lifecycle safety, and serialization; (2) after output pipeline and lock mechanics land, focusing on timing, RAM, and output correctness.
Communication L11. Known bugs block new features. No exceptions. A bug catalog is a blocking work item, not a reference document. Continuing to build UI and advanced features on top of known serialization bugs, state-machine bugs, and timing bugs compounds technical debt and makes the bugs harder to fix (more code depends on buggy behavior).

Phase 10.6 documented 12 confirmed bugs. Development continued into Phase 11 simplification, Phase 12 feature additions, hero page UI work, and Phase 14B architecture changes — all on top of the unfixed bugs. When Cluster A serialization bugs were finally fixed, the fixes had to account for 4 phases of code built on the broken foundation.

Action: Adopt a "bugs first" policy: any cataloged bug with severity ≥ medium blocks all new feature commits. The bug catalog is a living document; new bugs found during feature work are added before continuing.
Communication L12. Architecture decisions require explicit user approval before implementation starts. Major model/engine redesigns are not implementation details that can be decided by the implementing agent. They affect RAM budget, save/load compatibility, user-visible behavior, and migration cost. "I drafted a new architecture and started implementing it" is not an acceptable workflow.

Phase 14B seed+log model and Phase 16 flat-cell model were both drafted and partially implemented based on agent analysis. The parent/child model (Phase 7b) was built, debugged, hardware-verified, then replaced. User approval was implied by lack of objection rather than explicitly recorded.

Action: Any phase that rewrites the core model or engine must be preceded by a decision brief: what changes, why, RAM impact, migration cost, alternatives considered. Implementation starts only after user acknowledgment is recorded in the task wiki.
