Stochastic Track Atoms Inside the Performer Molecule

A plain-language decomposition of current stochastic features, plus the Westlicht Performer plumbing that makes them a real track instead of a standalone generator.

Core framing: stochastic features should be small internal atoms. The track is the molecule. A feature atom is only useful when it connects cleanly to Performer timing, patterns, routing, UI, storage, and hardware output.

Why This Exists

The current stochastic track has many user-facing controls. Complexity grows when each control brings its own timing path, storage path, UI truth, and playback branch.

The goal is not to invent new product vocabulary. The goal is to reduce the implementation to a few repeatable internal atoms, then connect those atoms through Performer track plumbing.

Decision Schematic: From Current C To The Next Sound

This is not sampled white noise at a random time. A C in the first octave is sounding because several decisions have already resolved: clock position, current loop slot, selected rhythm material, selected melody material, scale/root/octave/transpose, gate state, and output routing. The next sound is chosen by walking the same decision chain again.

0. Current state
What exists now C first octave is already on the output. The oscillator hears CV. The envelope/VCA hear gate.
Hidden state Current slot, elapsed ticks, current event duration, previous event cache, queue contents, RNG state, patience counters, jump offset.
User-facing context Pattern, track mode, source modes, First/Last/Size/Rotate, Divisor, Clock Mult, Scale, Root, Octave, Transpose.
1. Clock gate
Question Is it time for the next stochastic event yet?
Decision inputs Divisor, Clock Mult, current event duration, Sleep, Reset Measure.
Result If not enough ticks elapsed, C keeps sounding or releases by queued gate-off. If enough ticks elapsed, the next event is evaluated.
2. Frame
Question Which loop slot is being read?
Decision inputs First, Last, Size, Rotate, current slot.
Result The next slot is picked inside the active loop window. Rotate can read a different stored slot than the playhead position.
3. Source split
Rhythm source Rhythm Loop reads stored rhythm material. Rhythm Live rolls fresh rhythm now.
Melody source Melody Loop reads stored pitch material. Melody Live rolls fresh pitch now.
Overlap Loop/Live is not one switch. Rhythm and melody can disagree, so timing can repeat while pitch changes, or pitch can repeat while timing changes.
4. Rhythm choice
When is the next event? Note Duration, Variation, and Duration Tickets choose a duration slot.
Is it silent? Rest is a coin toss when rhythm is generated. In Loop mode the result is already stored until renewed or mutated.
How does it open? Slide and Legato are generated articulation decisions. Gate Length later turns duration into gate-off timing.
5. Melody choice
What pitch family? Pitch tickets, Range, Complexity, Bias, Spread, and Steps define the candidate pool and weights.
Which exact degree? A weighted draw picks degree/octave when melody is generated. Loop mode stores the draw. Live mode draws now.
What final CV? Scale, Root, Octave, Transpose, and Jump turn degree/octave into the output CV heard by the oscillator.
6. Repeat override
Question Should the next event reuse the previous emitted event?
Decision input Repeat is a playback-time coin toss.
Result If Repeat fires, the current slot's freshly chosen material can be bypassed by the previous event.
7. Filter
Question Does this event pass the loop thinning filter?
Decision inputs Mask sets how much survives. Tilt biases which duration ranks survive.
Result Failed Rest or Mask gives silence/gate-off. Passed material schedules CV and gate.
8. Hit count
Question How many hits does this one loop slot return?
Decision inputs Burst is chosen during rhythm generation. Burst Count and Burst Rate decide whether the result is zero, one, or several ordered hits.
Pitch rule Burst Pitch decides whether repeated hits reuse the first hit's pitch or ask melody generation for more pitches.
9. Schedule
CV queue The selected pitch becomes a CV event at an absolute tick.
Gate queue Gate-on and gate-off are queued from the evaluated hit list, Rest, Gate Length, and Legato.
Hardware The next audible note appears when queued CV/gate events reach the output tick.
10. Cycle end
When it wraps After Last, playback returns to First.
Evolution decisions Patience R/M can invalidate loop material. Mutate can permute or regenerate loop material. Jump can change octave offset. Sleep can delay later events.
Next cycle The next C might stay C, repeat, become another pitch, become silence, or be followed by more hits in the same evaluated step.

What this reveals: most user controls are not atoms by themselves. They are inputs to a few ordered decisions. The important simplification target is to make each decision produce one clear intermediate result, then feed the next decision.

Under The Hood: Engine Checklist

This is the same path from the engine's point of view. It starts with "C is currently sounding" and asks what data must be read, what formula or branch runs, and what gets written before the next audible output exists.

  1. Where am I? Current time, phase, input position, or linked position.
  2. Did a boundary happen? Whether playback crossed a line that requires a state transition.
  3. What is active now? The current piece of material after boundary handling.
  4. What does active material become? The evaluated runtime result after parameters, modulation, rules, and constraints.
  5. What output actions are required? The immediate or scheduled changes requested by the evaluated result.
  6. What output state is visible now? The current gate, CV, activity, and progress exposed to the rest of Performer.
  7. What must be remembered? Runtime state needed to answer the next tick correctly.
Segment Engine Question Data To Gather Formula Or Branch Writes / Output
1. Where am I? What tick is being processed? Performer transport tick, track engine runtime state. The engine runs once per clock tick. Queues are checked against absolute tick values. Any due gate/CV queue events are applied to physical outputs.
1. Where am I? How long is the current stochastic timing unit? Divisor, Clock Multiplier, global PPQN ratio. baseTicks = divisor * (CONFIG_PPQN / CONFIG_SEQUENCE_PPQN). Then divide by clock multiplier and round. Clamp to at least 1 tick. Runtime divisor used for duration and step timing.
2. Did a boundary happen? Has the current event duration ended? Current event duration, elapsed ticks since event start. If elapsed < eventDuration, do not choose a new event yet. C continues, or its queued gate-off may already have released the envelope.
2. Did a boundary happen? Is playback blocked before the next event? Sleep counter from previous cycle end. If sleep remaining is positive, decrement it instead of triggering a new step. No new material chosen. Existing output/queued releases continue.
2. Did a boundary happen? Did the active loop window change while playing? Fresh First/Last/Size, current pattern index, queued events. If active playhead is outside the new window, snap it to First. Clear stale gate/CV queues and force gate low. Playback stops referencing slots from the old loop length.
3. What is active now? Which loop slot is active? Current pattern index, First, Last, Size, Rotate. Clamp playhead into First..Last. Read index is optionally rotated inside that window. One read slot is selected. Playhead truth and read-slot truth can differ because of Rotate.
3. What is active now? Which source owns the active material? Rhythm mode, Melody mode, rhythm valid flag, melody valid flag, scale, root. Loop mode reads stored/generated material. If invalid, regenerate that domain before reading it. Live mode generates at trigger time. Valid rhythm and melody material are available for the selected slot.
3. What is active now? Does Repeat replace active material? Repeat probability, last emitted event, RNG. If last event is valid and random 0..99 is below Repeat, reuse the last emitted event. Either an override event is selected, or the engine continues evaluating the current slot.
4. What does active material become? What duration does active rhythm produce? Note Duration, Variation, Duration Tickets, RNG. Build weights around the Note Duration center. Variation widens the kernel. Duration Tickets multiply bucket weights. Weighted random draw picks one duration index. Duration index for this event.
4. What does active material become? How many ticks until the next event boundary? Duration index, runtime divisor. Look up the duration fraction. eventDuration = max(1, divisorTicks * numerator / denominator). New event duration is stored in engine runtime state.
4. What does active material become? Does rhythm request silence? Rest, RNG, rhythm validity. When rhythm is generated, Rest is a Bernoulli draw. In Loop mode the stored rest bit is read. Pre-filter rest state.
4. What does active material become? What gate behavior is attached to the material? Gate Length, Legato, Slide, RNG, event duration. Legato suppresses normal gate-off. Slide marks CV transition. Gate Length maps duration to a percent, with spread if the routed value is above zero, then floors to the audible minimum. Gate-off tick and slide flag are known if the event passes filters.
4. What does active material become? What pitch choices are legal? Scale, Range, Pitch Tickets, Min Degree, Max Degree, Steps. Enumerate legal scale degrees across the allowed range. Remove impossible or disabled ticket entries. Steps keeps only the strongest part of the pool. Candidate degree/octave list.
4. What does active material become? Which pitch is selected? Candidate pool, Complexity, Contour, Marbles Bias, Marbles Spread, previous degree, Pitch Tickets, RNG. Each candidate gets a weight from ticket strength, distance from previous degree, contour direction, and marbles bias/spread. Weighted random draw picks absolute degree. Event degree and octave. Previous-degree memory is updated.
4. What does active material become? What voltage target follows from selected pitch? Degree, octave, Scale, Root, track Octave, Transpose, Jump register. note = degree + (octave + trackOctave + jump) * notesPerOctave + transpose. Then scale maps note to volts. Chromatic root adds semitone voltage. Final CV target.
4. What does active material become? Does filtering silence the evaluated result? Mask, Tilt-derived rank, window size, rhythm/melody validity. If Mask is 100, pass. Otherwise pass when rank * 100 < mask * windowSize. Failed validity also silences. Final rest/pass decision.
4. What does active material become? How many hits does this evaluated step return? Burst, Burst Count, Burst Rate, Burst Pitch, duration ticks, RNG, candidate pitch machinery. If Rest/Mask fails, return zero hits. If Burst does not fire, return one hit. If Burst fires, return an ordered list of hits with offsets inside the evaluated step. The scheduler only needs the ordered list. hitList[]: each hit has tick offset, CV target, gate length, slide/legato flags, and optional UI/debug provenance.
5. What output actions are required? What if the hit list is empty? CV Update mode, final CV target, current tick. Empty list means no audible hit. Force gate low. Only queue CV if CV Update mode says rests should update pitch. Next output is Gate = low. CV either stays where it was or moves to the evaluated rest pitch.
5. What output actions are required? How is a hit converted to queue events? One hit from hitList[], current tick, step start tick. hitTick = stepStartTick + hit.offset. Queue CV at hitTick. Queue gate high at hitTick. If not legato, queue gate low at hitTick + hit.gateLength. Absolute CV and gate queue entries exist for this hit.
5. What output actions are required? How are several hits handled? Ordered hitList[]. Loop through hits in time order and apply the same queue formula to each one. Later hits are just more hits with later ticks. Queues contain one ordered hardware schedule for the evaluated step.
6. What output state is visible now? So what is the next gate/CV output? Earliest due CV queue entry, earliest due gate queue entry, current tick. When currentTick >= queued.tick, drain the queue entry. CV queue sets output voltage. Gate queue sets high or low. Concrete result: CV = X volts at tick T, Gate = high at tick T, then usually Gate = low at tick T + gateLength.
7. What must be remembered? Where does the playhead go next? Pattern index, First, Last. Increment pattern index. If it passes Last, wrap to First and mark cycle end. Next playhead position, cycle-ended flag.
7. What must be remembered? What history should UI and Repeat see? Evaluated hit list, rest/pass result, read index, final emitted CV/gate facts. Record recent output after evaluation, not before source choice. Repeat stores the emitted result, not merely the source slot. Recent-output history and last-emitted cache.
7. What must be remembered? At cycle end, does the future change? Patience R/M, Mutate, Jump, Sleep, Lock policy, RNG, source modes. Patience may invalidate loop domains. Mutate may permute or regenerate stored material. Jump may change octave register. Sleep may set a skip counter. Next cycle's source material or runtime modifiers may differ.
Explicit boundaries are checked by the outer engine tick. Nested boundaries appear after the evaluated material returns a hit list and are flattened into normal CV/gate queue actions.

The Performer Molecule

A stochastic track is not an isolated algorithm. It is one track mode inside Performer. That adds plumbing which is not optional.

Project/Pattern Pattern selection chooses which stochastic sequence is active. Copy/paste and save/load must preserve the model shape.
Clock Performer clock produces ticks. Divisor and clock multiplier turn user timing into tick intervals.
Track Engine Runtime state owns queues, current position, recent output, RNG state, patience counters, and gate/CV scheduling.
Routing Routable controls can be overridden by CV or internal modulation. The atom must read the routed value, not only the knob base.
Hardware Output Final CV and gate are consumed by oscillator, envelope, VCA, MIDI, or other downstream hardware.

What Each Atom Must Connect To

Performer Layer Question It Answers Why It Matters
Model ownership Is this pattern material, track-global control, or runtime state? Wrong ownership causes save/load lies, pattern switching bugs, and UI reading the wrong truth.
Clock order Does this happen on every tick, on event trigger, or at cycle wrap? Timing decisions cannot be scattered. They decide when the next note appears.
Routing Can this value be modulated? Routable controls need one read path so CV and base values do not diverge.
UI truth Does the page draw the stored material, generated output, or recent hardware output? Direct, Loop, Pitch, and Duration pages must not imply different realities.
Queue ownership Does this atom schedule gate/CV, or only produce material for the scheduler? Atoms that directly push queues multiply playback branches.
Resource budget Does this add bytes per pattern, per track, per engine, or globally? Per-pattern model bytes multiply fast. Engine bytes can affect all eight track engines through container size.
Track-mode integration Does this obey Performer reset, stop, pattern change, selected track changes, and output rotation? The stochastic track must behave like a Performer track, not a special side app.

Current Entanglement Smells

These are signs that a user feature is not a clean atom yet.

Feature Current Smell Cleaner Direction
Repeat It reuses previous material at playback time, so it interacts with RNG, mask rank, previous event cache, and burst playback. Make it either a generation-time decision or a clearly isolated playback transform.
Burst It is part stored recipe, part cache expansion, part playback branch. Treat it as one expansion atom that produces scheduled hits before the engine pump.
Live source It generates fresh material but also writes back into loop material for UI visibility. Separate live output history from loop storage.
Mask/Tilt Rank can live in stored event bytes, cache metadata, or recomputed runtime state. Pick one rank truth and make UI/engine read that.
First/Last edits The model changes immediately, but engine queues and current position may still point at old timing. Window edits must notify engine state immediately when the active track is playing.
Lock The UI has a toggle, but the engine currently ignores it after heap-cache removal. Lock should freeze one clear output truth, not a recipe plus runtime leftovers.

Recommended Internal Rule

Rule: a user feature should operate at one layer. If a feature touches model storage, cache building, queue scheduling, UI history, and cycle-end mutation at the same time, it is not an atom. It is a source of future bugs.

Layer Allowed Job Should Not Do
Generator Turn user controls and RNG into musical material. Drain hardware queues or depend on UI state.
Material representation Hold the generated loop truth in one format. Mirror another representation indefinitely.
Engine pump Schedule final CV and gate at ticks. Know why a hit exists unless necessary for timing.
Evolution Change material at cycle boundaries or explicit user actions. Patch already-scheduled hardware events invisibly.
UI Display stored material, generated output, or recent hardware output with clear intent. Use a source-tape view as a proxy for evaluated output without saying so.

Human-Time Playback Walkthrough

  1. User presses play. Performer clock starts producing ticks.
  2. The stochastic track reads pattern and track state: divisor, clock multiplier, source modes, first, last, size, rotate.
  3. The source atom decides whether rhythm and melody come from stored loop material or fresh generation.
  4. The time atom decides the event duration. That duration decides when the next main event can appear.
  5. The pitch atom chooses degree/octave. Performer scale, root, octave, transpose, and jump convert that into final CV.
  6. The articulation atom decides rest, slide, legato, and gate length.
  7. The hit-count decision returns zero, one, or several ordered hits for the evaluated step.
  8. The filter atom may silence material according to mask and tilt.
  9. The engine pump schedules CV and gate events at absolute ticks.
  10. At cycle wrap, evolution atoms may change the next cycle: patience, mutate, jump, sleep, NewR, NewM.

Useful Decision

The useful top-level decision is whether stochastic keeps one truth for evaluated musical material before it becomes gate/CV queue events.

Design Direction Effect Verdict
Keep source events plus cache plus playback fallbacks Preserves current behavior but keeps multiplying branches. Bad long-term
Delete source events but replace them with seed-only recipes Saves model RAM but weakens exact loop identity and lock semantics. Useful only if that product behavior is intended
Use one generated material format and make engine pump it Generator, engine, UI, and lock converge on one truth. Best architecture

Plain Test For Future Features

Before adding or changing a stochastic feature, answer these in order:

  1. Which atom does this belong to?
  2. Which Performer layer owns its state?
  3. When does it run: generation, trigger, tick, or cycle wrap?
  4. Does it affect stored material or only the next output?
  5. What does the UI draw: parameter, generated material, or recent output?
  6. What is the per-pattern, per-track, or per-engine RAM cost?
  7. What breaks if the feature is deleted?
