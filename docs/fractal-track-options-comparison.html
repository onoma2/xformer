FractalTrack — Implementation Options Comparison

Seven mechanisms evaluated against the spec's seven architectural necessities, with RAM/CPU notes and behavioral peculiarities. Scope is the full identity: two inputs, mutation, branches, ornamentation, looper behavior.

End-product-aware MVP path (Mirror with step grid)

The comparison below lands on Mirror with step grid as the substrate. This box is the actionable plan: an MVP that ships the smallest useful tape (capture, loop, lock, pattern-lens) while pre-wiring the structural hooks every later feature will need. Cost of pre-wiring is small (a few extra model fields, a few function boundaries in the engine); cost of not pre-wiring is rewriting the tick loop and the model layout when each later feature lands.

Principle

Structure for the full end product; implement only the MVP behavior. Every later feature has a known landing site at MVP — an empty function, a reserved field, a stub branch in a dispatch. The MVP runs the same engine the final product runs, just with most of the dispatch points being no-ops.

Serialization is done in full at MVP. Every known future field is declared in the model, initialized in clear(), written by write(), read by read() with sanitize/clamp, and sanitized in the bad-size branch — from day one. Inert fields hold defaults; UI doesn't expose them yet. This makes the MVP serialization layout the layout we live with through the whole feature buildout. No layout churn, no per-feature read/write additions, no "dev save files broke again." The MVP KPI is core functionality working and architecture sound; carrying a few hundred bytes of declared-but-unused fields is the right trade.

Model fields — what to reserve at MVP

Reserve cheaply now to avoid serialization layout churn later. Reserved fields hold sensible defaults and are inert; UI doesn't expose them yet. The ~1 KB cost across 8 tracks at worst case is far less than the cost of changing serialization layout multiple times during feature additions.

No reseed. FractalTrack is a child track — content originates from source, never from internal RNG. If the trunk is empty (never captured, or cleared), uncaptured cells default to the project root note (gate held off until capture or transforms touch them). Mutations and branches can then evolve that root-seeded baseline into something. This keeps the child-track identity pure: every cell's content has a parent justification, never a "fresh random fill."

Snapshot bank explicitly NOT reserved: the storage isn't a few bytes — it's another grid per snapshot. Reserving would force a worst-case allocation of N × grid-bytes inline now. Leave it as the one explicitly deferred feature; if added later, plan separately.

Engine structure — function boundaries to lock in at MVP

The MVP tick loop is structured so every late-stage feature has a named function or dispatch point to slot into. None of the structure is expensive; it just means writing helpers instead of inlining code.

tick(uint32_t tick) {
    // 1. Standard NoteTrack-style divisor + relativeTick math
    compute divisor, relativeTick;
    if (resetMeasure && relativeTick == 0) reset();

    // 2. Decide WHETHER to advance step. MVP: clockSource is always Internal.
    //    Future: when clockSource == External, read routedScan and dispatch differently.
    if (shouldAdvanceStep(tick)) {
        int newIndex = nextStepIndex(_stepIndex);

        // 3. Detect loop boundary. MVP: hook is empty. Future: mutation, patience, etc.
        if (crossedLoopBoundary(_stepIndex, newIndex)) {
            onLoopBoundary();
        }

        _stepIndex = newIndex;
        stepBoundary(tick, _stepIndex);
    }

    drainQueues(tick);
}

stepBoundary(uint32_t tick, int stepIndex) {
    if (!_lock) capture(stepIndex);
    play(tick, stepIndex);
}

// All read-position math collected into one place: rotate, loopPhase, fixed section phase.
int mapLogicalToRead(int logicalStep) {
    int windowSize = effectiveLoopLast() - loopFirst() + 1;
    int rotated = (logicalStep - loopFirst() + rotate()) % windowSize;
    // Apply loopPhase and the fixed 4-section phase lens here.
    // Future: branches apply by replacing this mapping with a transform.
    return loopFirst() + (rotated + windowSize) % windowSize;
}

// Source read isolated so two-source mixing can replace it without touching tick().
GateCV readResolvedSource() {
    return readSourceA();
    // Future: if (sourceB != -1) return mix(readSourceA(), readSourceB(), gateLogic, cvLogic);
}

capture(int writeIndex) {
    GateCV s = readResolvedSource();
    _trunk[writeIndex] = encodeCvAndGateLen(s);
    // If source gate is high, later falling edge before the next step updates gateLen.
}

play(uint32_t tick, int logicalStep) {
    int readIdx = mapLogicalToRead(logicalStep);
    StepWord value = _trunk[readIdx];
    // Future: value = applyBranchTransform(value, currentBranchPhase, readIdx);
    // Future: applyOrnamentation(value, logicalStep, tick);  // schedules extra events
    scheduleOutput(value, tick);
}

onLoopBoundary() {
    // Empty at MVP. Future hooks land here:
    //   - Proteus mutation roll (mutationProb, complexity)
    //   - Patience countdown / reseed trigger
    //   - Branch phase advance / trunk cycle counter
    //   - Octave shift mutation
}

// Future: int effectiveLoopLast() handles loopBars=0 (manual) vs loopBars>0 (derived from bars).
int effectiveLoopLast() { return loopLast(); }

// Future: nextStepIndex(curr) consults runMode for path-order (forward/reverse/pingpong/random).
int nextStepIndex(int curr) { return (curr + 1) % bufferLength; }

// Future: bool shouldAdvanceStep() returns false in External mode, true on edge-detected scan.
bool shouldAdvanceStep(uint32_t tick) { return relativeTick % stepDivisor == 0; }

What the MVP actually does (and only that)

  • Pick a source track (any lower-index TrackEngine). On every step boundary while not locked, the engine reads cvOutput(0) and gateOutput(0) from that engine, encodes CV plus gate length as a 16-bit word, writes the word to the current grid cell.
  • Plays back from the current grid cell at each step boundary: pushes a gate event and a CV event into the standard sorted queues at the current tick. Slide interpolation handled by update(float dt) via slideTime.
  • Loop window (loopFirst, loopLast), rotate (rotate), global phase (loopPhase), and fixed four-section phase constrain playback. Wrap at loopLast back to loopFirst. LoopMode=Once stops advancing after one pass; LoopMode=Loop continues.
  • Lock = stop capturing, keep playing. Unlocking resumes capture from whatever step is current.
  • Pattern switching changes loop window / rotate / loopMode / scale / root / divisor / etc. The grid itself survives across pattern changes (engine state, not model).
  • List UI exposes only the active fields. Reserved fields exist in the model and serialize but have no UI surface yet.

What's pre-wired but inert at MVP

  • Two-source mixing: sourceB + gateLogic + cvLogic reserved on track; readResolvedSource() is the swap site. Lands as a function-body change with no tick-loop edits.
  • CV-scan playback: clockSource reserved on sequence; shouldAdvanceStep() and nextStepIndex() are the dispatch sites. External mode is a new branch in those two helpers plus a routedScan read; no engine state-machine change.
  • Global and section phase: loopPhase and fixed sectionPhase[4] live on the sequence; mapLogicalToRead() is the math site. They are always part of the architecture, with neutral defaults and no user enable flag.
  • Bar-quantized loop length: loopBars reserved on sequence; effectiveLoopLast() helper is the derivation site.
  • Path / playback order modes: runMode reserved (already standard Performer); nextStepIndex() is the dispatch site. Adds forward/reverse/pingpong/random branches.
  • Recording variants (Latch / PunchIn): recordMode, punchMode reserved on track; capture() is the dispatch site. Adds two state branches.
  • Proteus mutation: onLoopBoundary() stub exists; engine has _rng field reserved (engine-state, zero RAM cost to add). Mutation params land on track when implemented.
  • Branches: play() calls a single-line trunk read at MVP; the line becomes a wrapped function call with branch-transform math later. Trunk phase counter + branch phase counter are reserved engine fields.
  • Ornamentation: play() calls an applyOrnamentation() stub at MVP; the stub pushes nothing extra. Late expansion adds anticipation/mordent/run events to the sorted queues — the queue infrastructure already exists.
  • Density / Tilt: No structural pre-wiring needed. Lands as a deterministic mask applied at play() before scheduling. Operates on cell ranking, doesn't touch capture or storage.
  • Snapshot capture (manual one-cell sample-and-hold): UI action calls a new snapshotCell(stepIndex) on engine; engine reads source immediately (not at next step boundary) and writes to the given cell. No model field needed.
  • Snapshot bank: The only feature with no clean pre-wired landing. Adding multiple buffers per engine requires either heap (each snapshot allocated on demand) or an extra inline buffer slot. Genuinely needs new structure later. Worth knowing.

Implementation phases

  1. Phase 1 — Model (with reserved fields). Create FractalSequence.h and FractalTrack.h with the full field set above. All write/read serialization slots present from day one. Track container entry, Routing target additions (including FractalRoutedScan for the future External clock source). STM32 sizeof probe.
  2. Phase 2 — Engine (with hooks but minimal behavior). FractalTrackEngine.h/.cpp with the function structure above. Inline step grid buffer sized at max bufferLength. Tick loop dispatches through the helpers; future-feature helpers return MVP defaults (no scan, no phase, no branches, no ornament, no mutation). Capture writes sourceA only. Lock works.
  3. Phase 3 — List UI. FractalTrackListModel.h exposes only the active params. Reserved params have no UI items. Source A clamp (must be lower-index track).
  4. Hardware verification gate. Assign a NoteTrack to sourceA, observe the FractalTrack output mirroring it at its own step rate. Toggle lock, observe playback freezes on the current snapshot. Switch patterns, observe the same grid sliced with different loop windows. Verify slide on playback, verify resetMeasure restart, verify pattern recall.

After this point, every late-stage feature is an independent landing inside an existing function boundary or reserved field — no structural rewrites, no tick-loop reshaping.

Order to add late features (suggested)

From cheapest to land to most behaviorally-rich:

  1. Path / playback order modes — extends nextStepIndex. Uses runMode field already standard in Performer.
  2. Two-source mixing — swap readResolvedSource. Adds the Vinx logic enum behavior.
  3. CV-scan playback (External clock source) — extends shouldAdvanceStep + nextStepIndex. Operational regime change; document that generative lifecycle pauses while scanning.
  4. Bar-quantized loop length — extends effectiveLoopLast.
  5. Recording variants (Latch, Once, PunchIn, recordTrigger) — extends capture.
  6. Snapshot capture action — engine entry point + UI action item. No model change.
  7. Sleep (between-loop silence) — first occupant of onLoopBoundary. Step-count silence after each loop pass.
  8. Density / Tilt — new mask in play.
  9. Proteus mutation + patience — also lives in onLoopBoundary alongside Sleep. Big behavioral feature, small code addition.
  10. Branches — wrap the trunk-read in play with a transform function. Add trunk/branch phase counters (already reserved engine state). When trunk cell is uncaptured, branches operate on project-root-note baseline.
  11. Ornamentation — fill the applyOrnamentation stub with anticipation/mordent/run scheduling.
  12. Snapshot bank — requires new structure. Plan separately.
  13. Visual page — buffer waveform, loop window markers, lock indicator.

The seven necessities

  1. N1 — Source reads with tick-order guarantee. Read another track's logical gate+CV on the current tick; relies on lower-index ordering.
  2. N2 — Small bounded trunk in engine state. Stored substrate that branches transform and mutation mutates; volatile, fixed-size.
  3. N3 — Step-boundary capture timing. Capture occurs at this track's step boundaries, not at tick rate and not at source events.
  4. N4 — Four-stage playback pipeline. Output = ornament(branch(sectionPhase(trunk-step))). Strict ordering at output time.
  5. N5 — Loop-boundary lifecycle event. A single named point in tick loop where mutation, patience, branch advance, octave shift fire.
  6. N6 — Track-level macro controls only. All behavior params are track-wide knobs, no per-step UI editing.
  7. N7 — Per-pattern lens, single-buffer body. Patterns provide different playback configs over the same captured trunk; no 17× buffer amplification.

Per-option detail

1. Tape — per-tick continuous recording Wrong scale

Capture source A's gate and CV every tick (192 PPQN or 1 kHz update) into a long buffer; play back per tick. Faithful reproduction at full source resolution.

Architectural alignment

●N1 reads source per tick
✗N2 buffer is large, needs heap
✗N3 captures per tick, not per step
◐N4 pipeline works but on dense data
◐N5 loop wrap at tick granularity
●N6 macros possible
◐N7 buffer survives pattern change but is huge
RAM
3 KB/bar at 120 BPM, 6 KB/2 bars. Heap mandatory at any reasonable loop length. Active-engine cap needed.
CPU
Capture itself is trivial. Branches at playback have to walk per-tick samples — a "reverse" branch reads a long buffer. Mutation picks a tick index among thousands.
Behavior
Captures source slides, micro-events, Teletype tick-level scripts, Tuesday micro-queues faithfully. Branches become coarse on dense data (transpose on every tick = a tone-shifted version of source). Ornamentation has no obvious step rate to attach to.

Why it loses fit: the trunk stops being a small fractal substrate. Branches and ornamentation lose their identity-defining "operate on N discrete steps" framing.

2. Mirror with step grid Reference fit

The track defines its own step rate via divisor. At each step boundary, sample source A (and optionally B mixed) and store one cell into a small fixed grid in engine state. Playback walks the grid.

Architectural alignment

●N1 reads source at step boundaries (still tick-order safe)
●N2 ~128 B grid inline
●N3 step-boundary capture is its core mechanic
●N4 step values feed branch math feed ornament
●N5 grid wrap = loop boundary, single point
●N6 every behavior knob is track-wide
●N7 grid survives pattern change; pattern carries loop window + divisor
RAM
64 steps × 16 bits = 128 B inline buffer. Each cell stores 11-bit CV, 4-bit gate length, and valid. Engine container under 500 B total.
CPU
One sample read + one encode per step. Branches apply a math transform to one cell per output. Ornamentation evaluated per tick around active step.
Behavior
Source is aliased to the track's step grid — fast Teletype events or sub-step Tuesday micro-queues collapse to one captured CV plus one gate length per step. This is the intentional design: the grid is the analytical unit. Locking freezes the grid; unlocking resumes capture. Pattern slots slice the same grid with different loop windows and section phase.

Why it fits: built for the spec. All seven necessities map onto its native mechanics. Smallest implementation, smallest RAM, simplest engine.

3. Logic-track style live mix (Vinx/Mebitek) Identity-incompatible

Each tick, read source A and source B, combine via gate logic (AND/OR/XOR/etc.) and CV logic (sum/avg/min/max/etc.), pass result to output. No storage, no looping.

Architectural alignment

●N1 reads sources every tick
✗N2 no buffer — nothing to transform or mutate
n/aN3 no captured grid
✗N4 pipeline collapses — branches have nothing to act on
✗N5 no loop boundary; output reacts continuously
●N6 gate-logic + CV-logic are track-wide macros
✗N7 nothing to lens
RAM
Engine state only, no buffer. ~30 B.
CPU
Two reads + one mix per tick. Trivial.
Behavior
Pure live combination of two sources. Output reacts instantly to source changes. No memory, no transformation over time. Stops the moment sources stop.

Why it loses fit: this is what LogicTrack would be (LogicTrack doesn't currently exist in the codebase but the pattern does in vinx-performer). Without storage the mutation/branches/ornament identity collapses. Useful as a separate track type but not as FractalTrack.

4. Performer-style CV recorder — steps live in model N6 + N7 violation

Arm record, then on each step the source's gate+CV gets written into the active sequence's step data (like NoteTrack's CV-record mode). Buffer == per-pattern sequence steps, persistent across save.

Architectural alignment

●N1 source reads work
✗N2 buffer in model, not engine
●N3 step-boundary capture is native
◐N4 pipeline works but trunk is sequence data
◐N5 loop boundary works
✗N6 per-step model state surfaces UI editing of captured steps — drifts toward step sequencer
✗N7 buffer × 17 patterns = massive RAM growth, each pattern recall swaps content
RAM
Per-pattern step data × 17 patterns. At 64 steps × 4 B × 17 = ~4.4 KB SRAM per track in model. Multiplies with track count.
CPU
Low. Same as NoteTrack-style record path.
Behavior
Each pattern slot is an independent recording. Switching patterns recalls different captured material entirely. Users can edit captured steps later. Save/load preserves the recordings. Feels like a NoteTrack with rec-input.

Why it loses fit: the spec commits to volatile single-buffer per engine (N7) and to no per-step UI (N6). This option breaks both. Worth considering as a separate track type for users who want persistent recordings, but it's not FractalTrack.

5. Sample & Hold — event-triggered capture Close cousin to mirror

Watch source A for gate rising edges. On each rising edge, sample source CV and write the (gate, cv, duration-since-last-event) triple into the next trunk slot. Capture rate follows source's event density.

Architectural alignment

●N1 source reads per tick (edge detection)
●N2 small event buffer in engine
◐N3 capture timing is source-event-driven, not step-boundary — alignment to track divisor is approximate
◐N4 pipeline works on events but "branches" on event lists behave differently than on a grid
◐N5 loop wraps when event buffer fills — irregular cadence
●N6 macros possible
●N7 buffer is engine-side, pattern provides lens
RAM
64 events × ~3 B each = ~200 B inline.
CPU
Trivial capture (edge detection). Playback walks events with elapsed-ticks counter, IndexedTrack-style.
Behavior
Captures source's note events faithfully — gate timing preserved. Sub-event source motion (CV slide between gates) is sampled once at gate-on. If source produces no gates, the buffer stays empty and the track outputs silence. Playback unit is "events" not "steps" — loop length is in event count, not bars. The musical feel is "loop the last N notes source played."

Why it almost fits: it preserves rhythmic identity of source better than step-aliased mirror. But N3 is the awkward one — capture is source-driven, so the track loses its own step rate as a concept. Branches and ornament still work but operate over events instead of steps, which changes their feel. Strong candidate if rhythmic fidelity matters more than analytical regularity.

6. Turing / shift register Wrong identity

Maintain a small fixed-length register (e.g. 32 cells). Every tick (or step), push source A's current gate+CV into one end; the cell falling off the other end is what plays now. Mutation = probability of bit-flip or new-roll during shift.

Architectural alignment

●N1 reads source per shift
●N2 register is the buffer
◐N3 shift happens at track-defined rate, but it's a sliding window not a step grid
◐N4 branches over a constantly-shifting buffer are conceptually awkward — what does "reverse" mean on a sliding register?
◐N5 loop boundary becomes "register wraps" — continuous not discrete
●N6 mutation prob and shift behavior are macros
◐N7 register state survives pattern change, but "loop window" doesn't apply naturally
RAM
32-cell register × 16 bits = 64 B inline. Among the smallest options.
CPU
One shift + one optional RNG roll per tick. Trivial.
Behavior
Output is a delayed echo of source with stochastic decay. Slow drift through the register creates loop-like repetition without a discrete loop. Mutation produces gradual divergence. Feels like Marbles Y/Z or Music Thing Turing — source-fed feedback delay with corruption. Not a "looper" in the traditional sense.

Why it loses fit: the spec's branches/ornamentation model assumes a discrete N-cell substrate with a clean wrap point. A shift register doesn't have either. Mutation is natural, but the rest of the identity (loop boundary, branches as transforms) doesn't map cleanly. Better as its own track type.

7. Granular scatter — positions over a ring Novel, partial fit

Maintain two storages. Storage A: small ring buffer of recent source ticks (a few hundred bytes, sliding window of last 1–2 seconds). Storage B: the trunk, which is a list of read positions into ring A — not actual sample values. At each step boundary, output reads ring A at the current trunk position. Branches reorder or transform the position list. Mutation moves positions.

Architectural alignment

●N1 source reads per tick into ring
●N2 trunk is the position list, small
◐N3 step boundary triggers position reads, but ring captures per-tick
●N4 branches transform position list, ornament decorates
●N5 trunk position list wraps = loop boundary
●N6 all behavior is track-wide
◐N7 ring is per-engine fine, but the position list per pattern would multiply
RAM
Ring buffer ~256 B (recent source) + position trunk ~64 B (64 positions × 8 bits each) = ~320 B inline.
CPU
Per-tick ring write is trivial. Position-list reads are simple. Branches that reorder positions are cheap (no value math). Mutation perturbs positions, also cheap.
Behavior
Output is recombinant collage of recent source — snippets of past source material played back in scrambled order. Source phrases become fragmented and rearranged. Mutation gradually drifts the position list. Branches like "reverse" reorder positions; "transpose" can still apply by shifting CV values at read time. Feels closer to Mutable Beads or a CV-domain Morphagene.

Why it almost fits: structurally clean — trunk is small, branches are zero-storage transforms over positions, ornament decorates output. The musical identity diverges from "child layer that loops source" toward "scatterer that collages source." More novel, less aligned with the spec's stated mental model. Possible later evolution if mirror-with-steps proves musically limited.

Decision rules

Read each rule as: this combination of wants forces or eliminates options.

If you want Proteus mutation + branches + ornamentation (the full identity)

Tape doesn't work — mutation on per-tick cells is musically inaudible (one tick is sub-millisecond); branches operating on thousands of ticks are wasteful; ornamentation has no step boundary to attach to. Logic doesn't work at all (no buffer). Turing doesn't work — math transforms on a constantly-shifting register are conceptually broken. Mirror, CV-recorder, S&H, and Granular survive. CV-recorder will break N7 next.

If you also want per-pattern lens (N7 — patterns slice the same captured material)

CV-recorder is out — its buffer is per-pattern model state, so each pattern holds an independent recording, not a lens over shared material. Mirror, S&H, Granular remain.

If you also want bar-quantized loop length / beat offset (loopBars, beatOffset from KD-15)

S&H is out — its trunk units are events (source gate fires), not steps, so "4 bars" doesn't map to event count. The musical loop length floats with source activity. Mirror and Granular remain. Mirror's bar-alignment is straightforward (steps × divisor = ticks); Granular requires an extra layer to align position-list traversal to bar timing.

If you also want predictable per-step ornamentation (mordents at step N-1, runs into step N)

Granular's positions don't have stable per-step musical identity — the cell at position 5 was sampled from a random ring offset, so "the note before this" varies across loops. Ornamentation that needs to anticipate "the next note" gets noisy. Only Mirror survives the full feature stack cleanly.

If you want only two-source mixing + path modes (mixer with sequence)

Logic gives you the mix but breaks on the loop, mutation, branches, ornament. Mirror gives you mix-at-capture + the rest. Mirror is the only path that lands both.

If you want collage / recombinant aesthetics (Morphagene / Beads feel)

Mirror produces faithful step-sampled playback — not collage. Granular is the natural fit because its trunk is positions over a ring, so "what plays at step 5" can be a snippet from anywhere in the recent source. But you give up bar-quantized loop, predictable ornamentation, and some recording-variant clarity. Different product.

If you want stochastic feedback / Turing-machine feel

Turing is the natural fit and nothing else in this list captures the same musical character (slow drift through a register with bit-flip corruption). But you give up branches, ornamentation, density, path modes, per-pattern lens — most of the FractalTrack identity. This is a separate track type, not a variation of FractalTrack.

If you want persistent recordings across save/load (each pattern = its own captured loop)

CV-recorder is the natural fit because its buffer lives in model state and is serialized with the project. Mirror and S&H lose their captures on power cycle. But CV-recorder gives up the per-pattern lens (N7) — each pattern is independent, not a slice of shared material. Different product again — closer to "NoteTrack with CV-input record mode" than to FractalTrack.

If you ever want tick-rate playback fidelity (capture every source gate, no aliasing)

Only Tape gives you this, and you pay for it with several KB heap per track, awkward mutation semantics, and ornamentation that has nowhere to attach. The spec's identity explicitly accepts step-rate aliasing — adding tick-rate fidelity later would force a rewrite of the trunk model and almost certainly a heap allocator path.

Cross-cutting note: scan playback and phase

These two features behave differently from the others. They aren't behaviors added to the trunk — they're alternative ways to compute the read position into the trunk. So their compatibility hinges on whether an option's trunk has a clean integer read index.

What scan mode actually does

Scan mode (KD-17) bypasses clock-based step advancement entirely. A routed CV maps directly to a step index in the trunk via floor-truncation, with edge detection firing output on step crossings. Same pattern DiscreteMap uses for External clock source. The clock-driven sequence loop is silent in this mode; the CV is the playhead. Useful for LFO-driven wobble, envelope-shaped phrase navigation, or sequencing the FractalTrack with another track's CV output.

What phase actually does

Phase is a read-position transform, not capture fidelity. Global phase (loopPhase) is a continuous fractional rotation of the playback head within the loop window. Fixed section phase splits the effective loop into four derived sections and bends read position inside each section. There is no user-facing enable flag: neutral values are identity, and an engine may fast-path neutral phase internally. Remainder cells are distributed front-to-back across the four sections.

Per option — both at once

  • Tape: both work cleanly. CV maps to tick offset, phase math operates on tick offsets. Tape's natural unit is the tick, so neither feature feels forced.
  • Mirror with step grid: both work cleanly. CV → integer step index for scan. Phase math is exactly CurveTrack's pattern, in step units. The grid is the natural addressable structure.
  • Logic: neither works. No playhead, no trunk to scan.
  • CV-recorder: both work the same way as Mirror — same step-grid abstraction, just persisted in model.
  • S&H: both partial. The trunk holds events with non-uniform durations, so CV-to-event-index is technically possible but musically uneven (scrubbing produces unequal time jumps). Phase offset has the same problem — fractional phase as a fraction of total loop time doesn't map evenly onto event indices.
  • Turing: neither works. There's no stable read index to address — the register shifts continuously, and "freezing" it to scan or phase-offset fights the Turing identity.
  • Granular: both work cleanly, possibly better than Mirror. The trunk is a position list, so scan maps CV onto a position (or directly onto a ring offset, skipping the list entirely). Phase offset rotates the position list. Granular was already a position-oriented model; these features fit natively.

Structural side effect of scan mode (applies to all options that support it)

Scan mode disables clock-driven advancement. That means features whose semantics depend on a natural loop-end event lose their scheduling moment:

  • Proteus mutation needs a "loop just wrapped" boundary to fire. In scan mode there's no wrap — the CV can sweep back and forth without ever crossing the conventional loop end. Mutation either has to fire on a different trigger (timer? trigger input?) or be disabled while scan is active.
  • Patience countdown measures elapsed loops. With no loop wraps, it stalls.
  • Predictive ornamentation (anticipations, runs toward the next note, mordents that reach for step N+1) can't predict what comes next when "next" depends on the user's CV. Two-step and four-step ornaments degrade to in-place behavior.
  • Branch phase counters (trunk for N loops → branches for M loops) lose meaning. Branches still apply their transform to whatever step the scan selects, but the trunk/branch cycle has no progression.

This isn't a per-option problem — it's a property of scan mode itself, and the same things would break in any substrate. The implication is that scan mode should advertise itself as a different operational regime: when scan is on, the generative lifecycle pauses and the track behaves as a pure CV-addressed wavetable readout. When scan is off, the full generative system resumes.

Phase has no such side effect — it's a pure read-position transform that composes cleanly with everything else.

Implication for the substrate choice

Mirror, Tape, CV-recorder, and Granular all support both features cleanly. Granular is interesting here because its native data shape (positions over a ring) is actually the natural representation of "CV-as-read-index" — scan mode is essentially what Granular does at all times. If scan playback is a centerpiece feature (not just a side mode), Granular has the structural advantage. If scan is one capability among many and Mirror's analytical grid suits the rest of the feature stack better, Mirror still wins on aggregate.

S&H's awkward marks on both features are worth weighing — it scored well on most other features but loses fit specifically on read-position-as-CV semantics. If scan is a must-have, S&H drops below Mirror.

Which option survives the longest "yes" chain

Walk the sixteen-feature list top to bottom, saying yes to each. Mirror is the only option that says yes to all sixteen without an awkward marker. Granular collects two awkward markers (per-pattern lens, recording variants), and that's it. S&H collects four awkward markers (branches feel different on events; scan playback is uneven; phase offset is uneven; bar-quantized loop is broken outright). CV-recorder collects one full break (N7). Tape collects three breaks (mutation, ornament, branches on dense data). Logic and Turing fall out within the first three features and never recover.

This is the structural argument for Mirror being the substrate even if some later feature seems initially better served by another option — every "later feature" that looks like a candidate for switching mechanisms ends up losing more than it gains because the other mechanisms break something earlier in the chain.

Recommendation

Mirror with step grid (option 2) is the only option that satisfies all seven necessities natively. Every other option breaks at least one architectural foundation — either by lacking the bounded trunk (logic), by storing too much (tape, CV-recorder), by capturing on the wrong axis (S&H, Turing), or by reframing the trunk into a non-grid structure (Granular) that strains the branch/ornament model.

Mirror is also the cheapest in RAM and CPU, and it inherits NoteTrack's proven timing pattern (divisor, relativeTick modulo, SortedQueue, slide-in-update, resetMeasure-on-bar) — no new engine timing model needs to be invented.

The runner-up by structural cleanliness is Sample & Hold (option 5) if rhythmic fidelity to source events outweighs analytical regularity of a fixed step grid. The runner-up by novelty is Granular scatter (option 7) if collage aesthetics are wanted later. Both could be considered as post-MVP track types or alternate modes within FractalTrack.

The remaining four options describe genuinely different products: a tape recorder (1), a live mixer (3), a persistent step recorder (4), a stochastic-feedback delay (6). They're worth knowing but they're not FractalTrack.
