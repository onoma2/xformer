Engine::update() — Current vs Proposed

Per-tick subsystem contention. T = tracks firing CvUpdate (≤8), K = queued ticks per call. Engine.cpp:65-214.

State comparison

Current

Structure

  • Inside the per-tick track loop, each firing track triggers three global recomputes: updateTrackOutputs + updateOverrides + routingEngine.update (:179-181).
  • Capped only by a per-track UpdateReducer<25ms> — frequency limit, not a structural fix.

Run counts

routing    = 1 + (T × K)
trackOut = (T × K) + 1
overrides = (T × K) + 1

Ordering defects

  • CV-route stale: loop runs updateTrackOutputs → updateOverrides; outputs consume _cvRouteOutputs before routing recomputes them → one update behind. (Root of the stability bug.)
  • Modulator lag: modulators tick after routing (:190); modulator-as-source is one tick behind.

Proposed

Structure

  • Hoist the three global passes out of the per-track loop → run once per tick, after all tracks have ticked.
  • The per-track UpdateReducer crutch becomes unnecessary.

Run counts

routing    = 1 + K
trackOut = K + 1
overrides = K + 1

Ordering fixes

  • CV-route current: reorder so routing / updateOverrides (CV router) run before updateTrackOutputs → physical outputs use current-tick routing. Fixes the stale bug structurally.
  • Modulator: tick modulators before routing, or accept + document the 1-tick latency.
Gate before any of this: confirm the track-linking intra-loop dependency. A track that links another (linkedTrackEngine) may read its source's freshly-composed output mid-loop — the likely reason the recompute is per-track today. If so, hoisting breaks linking. Dig the IO/layout/linking map first.

SWOT — hoist + reorder

Strengths

  • Removes the T× redundant global recompute — real CPU headroom on the hot path.
  • Fixes CV-route stale at the structural level, not a band-aid in one call site.
  • One deterministic pass order per tick — far easier to reason about.

Weaknesses

  • Requires resolving the track-linking dependency first (unknown until mapped).
  • Touches the audio hot path — regression risk, needs hardware audition.
  • No engine-timing test harness today; correctness rests on manual listening.

Opportunities

  • Frees CPU for heavier engines (Teletype, future Fractal) and more track types.
  • Retires the per-track UpdateReducer entirely.
  • Aligns with the routing-collapse work — both rationalise routing execution.
  • Could fold the duplicate suspended-path pipeline into one tail.

Threats

  • If linked tracks do need mid-loop fresh output, hoisting breaks linking silently.
  • Reordering modulators can shift modulation timing audibly.
  • Unaudited timing change risks subtle musical drift that unit tests won't catch.
