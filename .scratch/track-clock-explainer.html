Track clock computation — Free vs Aligned

Every track engine derives a phase that wraps 0 → 1 over its pattern. The two PlayModes differ only in how that phase is read from the clock.

The fork

Aligned

Phase from the integer master-tick grid. A step fires when relativeTick % divisor == 0.

Hard, transport-locked. The sub-tick fraction is ignored — phase holds flat between ticks.

Free

Phase from the continuous tickPosition(). A step fires when floor(tickPosition / divisor) changes.

Same musical grid — but sub-tick precise, so downstream phase math can be smooth.

The clock substrate

Master clock is 192 PPQN; steps run at 48 PPQN, so one sequence tick = 4 master ticks. tickPosition() = integer tick + the wall-clock fraction between ticks.

The phase every step-based track shares

divisor      = seqDivisor × 4 ÷ clockMultiplier   // master ticks per step
resetDivisor = resetMeasure × measureDivisor      // ticks per bar-reset (0 = never)
relativeTick = resetDivisor == 0 ? tick : tick % resetDivisor

patternPhase = (relativeTick mod patternTicks) / patternTicks    // 0 → 1, wraps

Shape of the 0 → 1 ramp

Aligned — integer-tick staircase Free — continuous tickPosition

Where 0 → 1 wraps: resetMeasure

The ramp resets at the pattern length, or — when resetMeasure > 0 — hard at the bar boundary (clearing queues + step state). With resetMeasure = 0 a track free-wheels on its own length and drifts against the bar.

Per track — how the 0 → 1 phase is built

TrackAlignedFreeshape
Note Fires on relativeTick % divisor == 0; sub-step gates/CV scheduled into SortedQueues (length, offset, ratchet, swing). Same grid via floor(tickPos/divisor). Note doesn't use the fraction.
Curve (relativeTick % divisor)/divisor — staircase, locked to grid. Phase accumulator _freePhase += ΔtickPos × (curveRate/divisor) — can run at a different rate; update(dt) interpolates for smooth 1 kHz CV.
Indexed No divisor-modulo. Per-step duration/gateLength in ticks; timers advance by ΔtickPos × clockMultiplier (always time-driven). phase = _stepTimer/duration; gate on while _gateTimer > 0. SyncMode::External starts steps on a routed CV rising edge.
DiscreteMap Internal ramp from relativeTick % periodTicks. Internal ramp from tickPosition; or External — ignores PlayMode & divisor, phase is the routed CV, stage fires on threshold crossing.
Tuesday Macro tick divisor-based like Note. Each boundary an algorithm emits sub-step events (trills/polys) into a micro-gate queue, each with its own cvTarget. Follows the shared grid; default Aligned.
Stochastic Event-triggered: a LUT decides the next event within the cycle. No continuous phase to smooth — Free vs Aligned only shifts the sub-tick resolution of the trigger instant.
PhaseFlux relativeTick → cycle position → deriveTickPosition() gives the active cell + a 0→1 stagePhase, sampled into per-stage curves every PPQN tick (≈ 2.6 ms ZOH @120 BPM). Uses the integer tick by design.
MidiCv Voice-allocated, not step-phased. The retrigger gate pulse rides the RTOS os::ticks() (~1 ms) — deliberately tick-quantized, outside the PlayMode model.
One-line model. Aligned = phase from the integer tick grid (hard, transport-locked). Free = phase from the continuous tickPosition() (sub-tick smooth). For most tracks they fire on the same grid and Free just adds resolution — only Curve and Indexed use Free to actually decouple the rate. DiscreteMap-External, PhaseFlux, Stochastic and MidiCv sit outside the plain Free/Aligned smoothness axis.
