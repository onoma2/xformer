Per-Type Param Tables — the internals

How U3 builds one table per track type, and why the name switch dies.

One idea: each track type owns a constexpr Row[] in flash. The engine picks the table once from track.trackMode(), then resolves paramKey inside it and calls the row's hook. The old 103-entry per-name dispatch collapses to an 8-way type→table map + a small key lookup — and every hook calls a concrete setter, so a dead route (routes but never writes) becomes impossible.

1 · One row

key
append-only numeric paramKey — never the array index
name
UI label (the only place the string lives)
range
{min,max} — the hook denormalizes the [0,1] value with it
flags
Continuous / Discrete / Structural / Inlet / Trigger
applyRouted
function pointer — this row's write, baked into flash

2 · Three hook flavors

The hook is where the old writeRouted switch-arm goes — but one arm per row, calling the concrete setter directly. scope.object is a Track*.

a. Track-direct — writes one field

// Octave on a Note track
void applyNoteOctave(const Scope &s, const Range &r, float n) {
    auto &t = *static_cast<Track*>(s.object);
    t.noteTrack().setOctave(round(denorm(r, n)), true);
}

b. Sequence-fanout — writes all 16 patterns

// Divisor lives on the Sequence; route writes every pattern so
// modulation is pattern-independent (the loop is a hook detail).
void applyNoteDivisor(const Scope &s, const Range &r, float n) {
    auto &t = *static_cast<Track*>(s.object);
    int v = round(denorm(r, n));
    for (int p = 0; p < CONFIG_PATTERN_COUNT; ++p)
        t.noteTrack().sequence(p).setDivisor(v, true);
}

c. Trigger — rising edge, not level already built

// engine resolves the edge against RouteState.gateMask, then
// calls the hook only on a low->high crossing.
void applyReset(const Scope &s, const Range &, float) {
    trackEngine(s.trackIndex).reset();   // fire
}

3 · Composition — a type's Row[] is the applicability matrix, literally

Table = Tier-1 universal ∪ backed Tier-2/2b ∪ backed Tier-3 ∪ Tier-4 own. Each type lists only the rows it backs; an inapplicable param simply has no row (can't be picked, can't no-op).

Note table

  • T1 Mute · Run · Reset · Divisor · ClockMult · Cv/GateOutRotate
  • T2 Scale · RootNote · Transpose · Octave · SlideTime
  • T3 FirstStep · LastStep · RunMode
  • T4 Rotate · 4× ProbabilityBias · LengthBias

Stochastic table

  • T1 Mute · Run · Reset · Divisor · ClockMult · Cv/GateOutRotate
  • T2 Scale · RootNote · Transpose · Octave · SlideTime
  • T3 none — own traversal, no step-range
  • T4 Mask · Tilt · Feel · Burst · Complexity · Contour · …

Same T1/T2 concepts, but the hooks differ: noteTrack().setOctave vs stochasticTrack().setOctave. So sharing is at the key/name/range level; the hook is always type-specific (the setters are unrelated functions).

4 · Table selection — the only switch left

const Table &tableForTrackMode(TrackMode m) {
    switch (m) {
    case Note:       return noteTable;
    case Curve:      return curveTable;
    case Stochastic: return stochasticTable;
    // … 8 arms
    }
}

The 8-way switch is irreducible: you must know a track's type to reach its setters. It runs once when a route resolves its scope — not per param, not per name. That is not the smear; the smear was the 103-entry name namespace, and it's gone.

Decisions (mine, leaning): (1) Explicit per-type Row[] over composition machinery — the "duplicated" T1/T2 rows are constexpr flash (no RAM), and the hooks must be type-specific anyway, so a shared-metadata + per-type-hook scheme buys indirection for little. (2) Shared paramKey registry for T1/T2/T3 — Divisor = the same key in every type's table, so a route survives a track changing type; T4 keys are per-type. Both append-only (F6). Open: whether sequence-fanout writing all 16 patterns stays the rule under the new format (it does today).
