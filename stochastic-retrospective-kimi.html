Stochastic Track — General Lessons for Future Projects

Context: Implementing a probability-driven sequencer track inside a RAM-constrained STM32 Eurorack firmware.

Commits: 107 across 6 days (May 17–22, 2026).

Focus of this document: Generalizable lessons extracted from the specific mistakes and challenges, intended to guide future projects.

1. Project Overview (Brief)

Objective: Port and extend a stochastic sequencer track into Performer's native architecture — model, engine, and UI — while staying under strict RAM/CCMRAM gates.

Key challenge: The project went through rapid architectural iterations (V3 → V4 → V5, Phases 7–16), multiple adversarial reviews, and several hard-reboot bugs before stabilizing.

Outcome: Hardware-stable implementation with four hero UI pages, deterministic generators, and a chosen future direction (seed+log architecture). The real value for the organization is the process lessons below.

2. General Lessons for Future Projects

Communication & Process

1. Maintain exactly one living specification; archive everything else.

This project accumulated overlapping documents (V3, V4, V5, Phase 7 through 12) with contradictory definitions for the same terms. Implementation drifted depending on which document the current developer read last. For future projects: designate one file as the contract. Move all outdated versions into an archive/ folder and prepend explicit deprecation notices. If a term changes meaning, rename it everywhere or the old spec becomes a landmine.

2. Ground-truth the model before designing the visual layer.

Eleven poetic UI preview variants were rendered in Python before the canonical control vocabulary was extracted from existing codebase patterns. The beautiful previews assumed controls that did not yet exist in the real C++ structs. For future projects: require header-file field names, ranges, and defaults to be frozen before any visual design begins. The UI layer must map 1:1 to real struct members, not to aspirational concepts.

3. Write acceptance criteria for feel changes; do not rely on verbal agreement.

Major audible behavior shifts (baked repeat vs floating, per-cycle Live regen, project migration loss) were accepted in design sessions but not always recorded before code moved forward. For future projects: any change that alters user-facing behavior must have a written checklist that is explicitly signed off before implementation starts. Verbal acceptance evaporates when the person who gave it is no longer on the thread.

4. Cross-reference architecture decisions in source files, not just wikis.

Critical fixes (Container destructor leak, engine rebuild order) were logged in the task wiki, but subsequent developers rediscovered the same bugs independently because they edited the source without reading the wiki first. For future projects: add a one-line comment in the relevant header file pointing to the wiki decision. Wikis are passive; source comments are active.

5. Treat duplicate or ambiguous labels as bugs, even if the code compiles.

After renaming "Character" to "Complexity," both the old enum entry and the new entry remained in the UI list model, both displayed as "Complexity." Users could not distinguish them, and bug reports became impossible to interpret. For future projects: every rename commit must include a grep pass for the old label in list models, footer strings, and documentation. If two controls share a label, one must change or one must be deleted.

Architecture & State Ownership

6. Const-cast in an engine is an alarm bell for state ownership.

When the engine task cast away constness to write generated events back into the model sequence, the architecture lost its invariant. The engine and UI tasks then raced on the same packed memory without a write lock. For future projects: treat any const_cast that enables mutation as a code-review stop. Either move the output to engine-owned runtime state, or define a locked write path with clear ownership. State ownership is not a style issue; it is a correctness issue.

7. Pattern identity must follow the cache.

The lock replay buffer was keyed only by array index. Changing project patterns while Lock was active replayed cached events from the previous pattern at the same index, producing musically random output. For future projects: any cache that outlives a context switch needs an identity tag (pattern ID, scene ID, bank number) or must be explicitly invalidated on every transition. Caches without identity are time bombs.

8. Variant storage must type-erase and invoke destructors.

The project's Container<A,B,C> placement-new'd the next occupant without calling the previous one's destructor. A heap-allocated lock buffer leaked ~1.8 KB on every track-mode switch, exhausting the STM32 heap and causing hard reboots. For future projects: any union, variant, or type-erased buffer must store a destructor thunk and invoke it on replacement and on scope exit. This is generic infrastructure, not a feature-specific concern.

9. Enforce tick-order invariants at the architecture level.

A single-line fix — moving updateTrackSetups() before clock-event processing — eliminated a hard-reboot bug where an engine accessed model memory that had already been repurposed. For future projects: document the invariant explicitly ("engine containers always match model trackMode before any engine code runs") and assert it in code if possible. On embedded systems, ordering bugs become reboots, not exceptions.

UI & Embedded Safety

10. Type-specific pages must guard every callback locally.

Multiple draw/LED/input paths called selectedTrack().stochasticTrack() without verifying the track mode. Project load and mode mutations outside the normal navigation path left stale pages alive, leading to type-specific accessors on wrong memory layouts. For future projects: every page that calls a type-specific accessor must early-return if the mode mismatch is detected. Do not assume the navigation layer will always replace the page first.

11. Never expose a control that has no engine branch.

PlayMode (Aligned / Free) was persisted and shown in the config UI, but the engine never branched on it. Users saw a control with no audible effect, creating false expectations and wasting support time. For future projects: if a parameter is not wired to behavior, hide it from the UI and do not serialize it. An unimplemented control is worse than a missing one because users will report it as broken.

Testing & Math

12. Serialization symmetry must be proven by a round-trip test, not by eye.

Three pattern-defining fields were initialized by clear() but never written or read. Save/load silently dropped generative identity. Conversely, transient engine state (_lock) was serialized, violating the ownership contract. For future projects: require a TDD round-trip test for every new model type: set non-default values, write, clear, read back, assert equality. This test is cheaper than a hardware debugging session.

13. State machines need explicit reset and restart coverage.

The ticket-mode timing state worked during normal playback but was uninitialized on resetMeasure() and restart(), causing up to 384 ticks of silence or machine-gun retriggering. For future projects: every state variable introduced for a mode must be explicitly initialized in reset(), restart(), and clear(). Reset paths are where state machines break; they deserve their own test cases.

14. Audit probability kernels for integer-division cliffs.

Five separate picker kernels used knob / N integer division, creating silent cliffs at low knob values. A user setting a knob to 10 got audibly identical output to a knob at 0 because the division truncated to zero. For future projects: grep for division by constants inside probability kernels. Either replace with scaled multiplication or document why floor behavior is musically intended. Probability math that collapses to zero at non-zero settings is a UX failure.

15. Default values must exercise the feature, not collapse it.

A default variation of 0 collapsed the duration picker to a single deterministic slot, and all Bernoulli probabilities defaulted to 0. Pressing "New Pattern" at defaults produced identical output, so users thought the feature was broken. For future projects: review every new parameter's default. If zero means "feature is effectively off," raise the default to the minimum audible setting and document why.

Memory & Hardware

16. Heap allocation on firmware is a last resort, not a convenience.

The heap-allocated lock buffer saved CCMRAM on paper but introduced a crash vector under sustained load. It was later removed after a hardware failure. For future projects: prefer static pools, engine-owned CCMRAM with explicit size gates, or bounded project-owned pools. If heap is unavoidable, expose allocation failure through UI feedback and disable the feature when unavailable. Silent failure on RAM-critical firmware is unacceptable.

17. Macro controls must be one-to-one with single fields.

Original L1 macros wrote through to multiple sequence fields in lockstep, clobbering explicit user edits at lower levels. The redesign made each macro write exactly one field, with engine-clever derivation handling the sophistication. For future projects: macros that fan out to multiple fields create hidden coupling and destroy user agency. One knob, one field. Sophistication belongs in the engine, not in the edit path.

18. Queue-based scheduling needs envelope semantics for overlapping events.

Independent gate-ON/OFF pushes into a sorted queue created truncation when a parent OFF preceded a child ON. Burst children were silently cut to 1-tick gates. For future projects: when scheduling multiple related events into a queue, define whether the parent event's OFF is authoritative or whether child events extend the envelope. Document the tie-breaking rule for same-tick collisions.

Label audit on every rename Add a CI script or pre-commit check that greps for old labels in *ListModel*, *Page*, and manuals/ after any enum or field rename. Medium
